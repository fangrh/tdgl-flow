# Real-time I-V Scanner & Progress Refactor

## Goal

Remove the viewer's dependency on `iv.json` sidecar and `simulated` progress label. Replace with a background thread that computes I-V directly from frame running_state data, and a timing-based progress bar driven by the known solve schedule.

## Background

Currently the viewer has two problems:
1. **Progress label "simulated"** relies on `refresh_index()` to detect new frames via viewer-index.json etag changes. When refresh fails silently, the label freezes.
2. **I-V data** comes from `iv.json`, which is only generated after the simulation completes. During a running simulation, there is no I-V information.

Both problems stem from depending on sidecar files that may not exist during a live simulation.

## Design

### 1. Remove "simulated" label, use timing-based progress

The `manifest.json` (available from simulation start) contains `timing_steps` with `solve_time` and per-step time ranges. The progress bar can show `t=X / solve_time` directly from the time slider value — no server-side data needed.

**Changes:**
- `progress_label` in `widget.py`: show `t={time_slider.value:.1f} / {solve_time:.1f}` instead of `simulated: {latest_t:.1f} / {solve_time:.1f}`
- Remove `latest_frame_time()` call from `_live_refresh`
- `_live_refresh` still calls `refresh_index()` to pick up new frames for playback, but no longer updates the progress label

### 2. Background IV scanner thread

A dedicated Rust thread scans frames sequentially from frame 0 onward, reading `running_state/mu` and `running_state/dt` via Range requests using offsets from viewer-index.json.

**Scanner behavior:**
- Starts automatically when `open()` is called
- Reads frames in order: 0, 1, 2, ...
- For each frame, uses `frame_rsmu_offsets[i]` and `frame_rsdt_offsets[i]` to Range-read running_state data
- Computes voltage via existing `compute_frame_voltage(rsmu, rsdt)`
- Groups voltage samples by timing step (using cumulative time from `frame_times[i]`)
- Stores results in a shared `IVScanResult` struct behind `Arc<Mutex<>>`
- Stops when it reaches the last known frame; resumes when `refresh_index` discovers new frames
- Python polls `get_iv_progress()` for the scan head position and I-V data

**Data flow:**
```
viewer-index.json offsets
       ↓
IV scanner thread → read_range(rsmu_offset, rsmu_size)
                  → read_range(rsdt_offset, rsdt_size)
                  → compute_frame_voltage()
                  → group by timing step
                  → Arc<Mutex<IVScanResult>>
       ↓
Python widget polls get_iv_progress()
       ↓
I-V plot renders progressively
```

### 3. I-V progress indicator

The progress bar shows which time range has been scanned:
- Already scanned region: colored (I-V data available)
- Not yet scanned: grayed out
- Scan position updates as the background thread progresses

**Implementation:** The `get_iv_progress()` method returns:
```json
{
  "scanned_frames": 500,
  "total_frames": 8951,
  "scanned_time": 3500.0,
  "solve_time": 60000.0,
  "steps": [
    {"step_idx": 0, "je": 0.0, "v_avg": 0.5, "n_samples": 50},
    {"step_idx": 1, "je": 0.2, "v_avg": 0.8, "n_samples": 50}
  ]
}
```

### 4. Remove iv.json dependency

- `load_iv_sidecar()` no longer called in `open()`
- `iv.json` still generated by runner for backward compatibility, but viewer ignores it
- I-V data always comes from the live scanner

### 5. What stays the same

- `viewer-index.json`: unchanged, continues providing byte offsets for Range requests
- `refresh_index()`: unchanged, still polls for new viewer-index versions
- Frame rendering: unchanged, still uses psi/mu offsets for 2x2 plot
- `manifest.json`: still used for run listing, timing steps, solve_time
